# Правила для ассистентов (репозиторий `hiddify-app`)

## Актуальная задача

Текущий приоритет: архитектура и реализация MASQUE в `hiddify-core` (форк `sing-box`) с двумя новыми endpoint-ами:

- `masque` (самостоятельный generic endpoint);
- `warp_masque` (Cloudflare-совместимый endpoint на том же MASQUE-ядре).

Legacy `warp` должен оставаться параллельно и без изменений.

Подробная архитектура: `hiddify-core/docs/masque-warp-architecture.md`.

## Обязательные инварианты

- Не менять поведение и контракт legacy `warp`.
- Не делать скрытую миграцию legacy `warp` на новый runtime.
- Новая функциональность добавляется только через новые endpoint-типы (`masque`, `warp_masque`).
- Архитектура должна соответствовать endpoint-модели `sing-box` и его data plane (`route`, `tun`, `ListenPacket`/`DialContext`).
- MASQUE-ядро должно быть единым и переиспользуемым между `masque` и `warp_masque`.

## Правило parity для авторегистрации

Семантика `warp_masque` по авторегистрации обязана соответствовать фактической семантике текущего WARP в форке:

- если в текущем WARP регистрация встроена в endpoint/core path — так же делается в `warp_masque`;
- если регистрация внешняя — в `warp_masque` не встраивается в core.

На текущем этапе parity фиксирован на встроенной регистрации (см. архитектурный документ).

## Chaining и режимы MASQUE

- В архитектуре сразу закладывать single-hop и multi-hop (chain graph/policy).
- Даже если реализация идет поэтапно, API и конфиг-контракты должны изначально учитывать chaining.
- Избегать протокольных костылей, не описанных архитектурой.

## Принципы изменений

- Изменения точечные, с обратной совместимостью.
- Приоритет: корректность, предсказуемость поведения, отсутствие регрессий.
- Не добавлять фичи вне текущей задачи.
- Перед завершением — проверка сборки/тестов в затронутых частях.
