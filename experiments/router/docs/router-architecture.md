# L3 Router — целевая архитектура (звезда / статические «пиры»)

## Назначение

**l3router** — L3-коммутатор на VPS (**хаб, звезда**): сырые IP-датаграммы между клиентскими сессиями **без криптографии в движке**, по смыслу как **WireGuard AllowedIPs + пиры**, но транспорт и шифрование — **внешние** (любой inbound/outbound sing-box).

Это **не mesh-сеть** и **не** обязательный runtime control plane.

## Топология

- Один **центральный** sing-box (хаб).
- Каждый клиент — **одна логическая сессия** (идентификация как у multi-user inbound: user / тег).
- Маршрутизация **только через хаб** (нет прямого пира-пир без VPS в этой модели).

## Статический маршрут = статический «пир»

Каждый маршрут задаётся **в конфиге** (как пир в WG), не меняется в рантайме для базового сценария:

- **Префикс(ы) клиента** — адресное пространство, с которого этот клиент имеет право слать (аналог **AllowedIPs / source**).
- **Куда форвардить** определяется **полями IP-пакета**: прежде всего **dst** (LPM по таблице «какой префикс принадлежит какому маршруту/сессии»), плюс **ingress ACL** по **src** (анти-спуф).

Идентичность отправителя берётся из **привязки сессии ↔ маршрута** при установлении туннеля (как «этот ключ = этот пир», только без ключей — из контекста sing-box).

## Data plane (минимум)

1. Пакет приходит с inbound в l3router с **известной сессией**.
2. Определяется **ingress-маршрут** по привязке сессии.
3. Проверка **src ∈ allowed** для этого маршрута.
4. **FIB lookup по dst** (longest-prefix) → выбор **egress-сессии** (другой «пир»).
5. Отправка в outbound сессии пира **в порядке пакетов**, без лишних копий где возможно.

## Control plane

Для целевого сценария **достаточно статического JSON** (как список пиров в WG).  
Отдельный HTTP API для постоянной смены «владельцев» и динамических маршрутов **не является частью этой модели**.

## Отношение к текущему коду в дереве

Реализованный **MemEngine + owner + runtime upsert через Clash API** — **отклонение** от целевой звезды: ориентир на динамику и политики, ближе к **варианту с control plane**, а не к минимальному «WG без крипты».

Целевое направление работ: **упростить до статических маршрутов и dst-ориентированного FIB**, убрать обязательность control-plane пути для базовой эксплуатации.
